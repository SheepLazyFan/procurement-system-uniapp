场景 B：智能文档生成 (The Technical Writer)
适用时机：项目交付前，或需要为遗留代码补充文档时。根据 Swimm 的理念，文档应解释“业务逻辑”而非仅仅翻译代码。

**[Role / 角色]**
你是一位专注于开发者体验 (DX) 的高级技术文档工程师。

**[Context / 上下文]**
请阅读用户的项目代码。

**[Task / 任务]**

**[Content Structure / 内容结构]**

场景 C：遗留代码维护与重构 (The Modernizer)
适用时机：接手老旧代码（Legacy Code），或者需要优化“屎山”代码时。
1. 解释晦涩代码 (Gemini/Copilot) 当遇到看不懂的复杂逻辑时：
**[Task / 任务]**
请像给初级开发者讲课一样，解释这段代码的业务逻辑。
1. 它在做什么？(What)
2. 为什么要这样写？(Why)
3. 这里的 `magic number` (魔术数字) 或复杂的正则表达式代表什么含义？
4. 存在哪些潜在的性能隐患或安全风险？
2. 代码现代化重构 (Copilot Chat) 在 IDE 中选中老旧代码块（例如使用了回调地狱的 JS 代码）：
**[Task / 任务]**
重构这段代码，使其符合现代 {语言版本，如 ES6+} 标准。

**[Refactoring Goals / 重构目标]**
1. **语法升级**：将 `var` 改为 `const/let`，使用箭头函数。
2. **异步处理**：将回调函数 (Callbacks) 重构为 `async/await` 模式。
3. **可读性**：提取复杂逻辑为独立函数，并添加类型注解 (TypeScript)。
4. **安全性**：如果存在 SQL 注入或 XSS 风险，请修复。

**[Constraints / 约束]**
保持原有业务逻辑不变（功能等价性）。
