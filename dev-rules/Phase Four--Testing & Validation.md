步骤 0：定义测试策略 (Gemini Web)
适用时机：在开始写代码前，先让大脑（Gemini）想清楚要测什么，防止漏测。
请将以下内容复制给 Gemini：
**[Role / 角色]**
你现在是我的 QA 负责人 (Head of Quality Assurance)。

**[Context / 上下文]**
我们要为 `TASKS.md` 中的任务：编写测试计划。
相关的类型定义 (Types) 需要向用户索要，否则不进行下一步操作。

[Task / 任务] 请列出该功能必须覆盖的测试场景列表。请包含：
1. 快乐路径 (Happy Path)：最标准的用户操作。
2. 边缘情况 (Edge Cases)：例如空输入、极大数值、负数、网络延迟等。
3. 安全边界：例如非法字符注入（如果适用）。
[Output / 输出要求] 请不要写代码，只输出一个 Markdown 列表，描述测试意图。例如：“测试当购物车为空时，总价应为 0”。

#### 步骤 1：红色阶段 - 编写失败测试 (Copilot Chat)

**[Mode / 模式]**
TDD Red Phase (只写测试)

**[Context / 上下文]**
请参考当前打开的 `types.ts` 文件。询问用户：需要对哪些任务进行测试？否则不进行下一步操作。

**[Task / 任务]**
请编写一个 Jest/Vitest 测试文件。
1.  遵循 **AAA 模式** (Arrange-Act-Assert)。
2.  测试描述 (describe/it) 必须清晰表达业务意图。
3.  **断言 (Expect)** 必须严格匹配预期结果。
4.  **重要**：只输出测试代码，不要输出实现代码。确保这个测试现在运行是**失败**的 (Red)。

步骤 2：绿色阶段 - 最小化实现 (Copilot Chat)
适用时机：当用户运行测试并确认它失败（报错）后。

**[Mode / 模式]**
TDD Green Phase (只写实现)

**[Context / 上下文]**
请查看当前打开的测试文件（要求用户给出报错信息，否则不进行下一步操作）。

**[Task / 任务]**
请编写/修改业务代码，使其**刚好**能通过上述测试。
1.  **KISS 原则**：Keep It Simple, Stupid。不要过度设计，不要包含额外的功能。
2.  **目标**：让测试变绿 (Pass)。
3.  **输出**：只提供需要修改的代码片段。

步骤 3：重构阶段 - 优化代码 (Copilot Chat)
适用时机：测试通过后。千万不要跳过这一步，这是防止代码变成“屎山”的关键。

**[Mode / 模式]**
TDD Refactor Phase (重构)

**[Task / 任务]**
现在的代码已通过测试。请作为资深工程师审查当前实现：
1.  **代码异味 (Code Smells)**：是否有重复逻辑？命名是否清晰？
2.  **结构优化**：是否可以将长函数拆分为辅助函数？
3.  **类型安全**：是否有 `any` 类型需要优化？

**[Constraints / 约束]**
请给出重构后的代码。
**绝对禁止**改变代码的外部行为（Public Behavior）。重构后，现有的测试必须依然通过。

步骤 4：自动化测试生成与自我修复 (Copilot / Gemini)
适用时机：当你有一个遗留函数没有测试，或者测试报错你看不懂时。
场景 A：为旧代码补充测试 (Copilot)
**任务**：为当前选中的函数生成单元测试。
**要求**：使用表格驱动测试 (Table-Driven Tests) 的方式，覆盖至少 5 种不同的输入组合（包括空值、非法值）。
场景 B：测试报错修复 (Gemini Web) 如果 Copilot 无法解决复杂的测试错误，将错误日志和代码投喂给 Gemini。
**上下文**：我正在运行测试，遇到了以下错误日志。
**代码**：{粘贴相关代码}
**错误日志**：{粘贴 Error Log}
**任务**：分析测试失败的根本原因。是因为业务逻辑错误，还是测试用例写得不对？请给出修复方案。

# Testing Guidelines (全局测试规范)
当被要求编写测试时，请始终遵循以下标准：
1. **框架**：默认使用 Jest (或用户选用的框架)。
2. **命名规范**：测试描述应采用 "Should return X when Y given" 的格式。
3. **隔离性**：每个测试必须独立运行，Mock 所有外部依赖 (API, DB)。
4. **AAA 模式**：代码结构必须分段为 // Arrange, // Act, // Assert。
5. **红绿循环**：如果在写测试，不要同时生成实现代码。
