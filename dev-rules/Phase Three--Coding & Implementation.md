场景 A：初始化/生成核心模块 (The Builder Prompt)
适用时机：编写一个新的组件、服务或函数时。

**[Context / 上下文]**
严格检查上下文，需清楚该项目需要的技术栈，并把你总结出的技术栈向用户报告。
请参考当前打开的 `types.ts` 文件中的数据定义。

**[Task / 任务]**
实现 `TASKS.md` 中的任务，给出你实现的具体的原子任务，例如：1.2 实现用户注册服务函数。这个时候强烈要求用户给出“Task.md”文件，才能进行下一步操作。

**[Constraints / 编码约束]**
1. **类型安全**：严格遵循 TypeScript 接口定义，严禁使用 `any` 类型。
2. **防御性编程**：
   - 对所有输入参数进行 Zod 验证（如果适用）。
   - 必须包含 try-catch 块，并返回标准化的错误对象。
3. **原子性**：只生成当前任务所需的代码，不要过度设计。
4. **注释**：为复杂的逻辑添加简短的 JSDoc 注释。

**[Style / 风格]**
请生成 {函数式/面向对象} 风格的代码，保持代码简洁（Clean Code）。
设计逻辑：
• 引用类型定义：明确要求参考 types.ts，利用 Copilot 的上下文感知能力减少幻觉,。
• 防御性编程：强制要求错误处理和输入验证，这是 AI 生成代码最容易忽略的地方,。

场景 B：测试驱动开发 (The TDD Cycle)
适用时机：编写关键业务逻辑前，确保代码稳定性。遵循“红-绿-重构”流程。
步骤 1：生成失败测试 (RED)
**任务**：为上述 {功能名称} 编写单元测试（使用 Jest/Vitest）。
**要求**：
1. 仅编写测试用例，不要实现功能代码。
2. 包含 1 个“快乐路径”（Happy Path）测试。
3. 包含 2 个“边缘情况”（Edge Cases）测试（例如：空输入、网络超时、非法格式）。
4. 确保测试当前是失败的（Red）。

步骤 2：生成最小实现 (GREEN)
**任务**：编写通过上述测试所需的**最小量**代码。
**约束**：
1. 不要优化性能，首要目标是让测试变绿（Pass）。
2. 严格匹配测试用例中的断言逻辑。
步骤 3：重构与优化 (REFACTOR)
**任务**：现在测试已通过，请重构代码以提升可读性。
1. 提取重复逻辑到辅助函数。
2. 优化变量命名，使其更具语义化。
3. **保持测试通过**：确保重构不破坏现有逻辑。
